# 签字报表确认自动化工具 项目提报材料

## 一、项目概述

**项目名称：** 签字报表确认自动化工具  
**数据规模：** GF美区员工考勤签字报表（项目可复用其他区域）
**核心功能：** 签字报表数据自动核查与风险分析、飞书机器人自动化发送考勤确认消息、收集员工回复、生成进度报告与数据校验  
**项目状态：** 区域试用中

---

## 二、原始痛点分析（Situation & Task）

### 2.1 原有工作方式（AI分析）

| 环节         | 原有方式                | 预估人力投入            |
| ---------- | ------------------- | ----------------- |
| 发送确认消息     | 人工逐一手工发送飞书消息        | 304人 × 3分钟 ≈ 15小时 |
| 追踪确认状态     | 人工翻阅聊天记录、Excel登记    | 每轮追踪约2-3小时        |
| 汇总确认进度     | 人工汇总回复、统计各状态人数      | 每次汇总1-2小时         |
| 数据校验       | 人工对比签字报表与打卡/补签/休假记录 | 1人次 × 8小时         |
| **单轮完整流程** | **全手工**             | **约26-28小时**      |

### 2.2 核心痛点

- **重复性劳动密集**：全美区需要各地BP逐一核对数据，数据裁剪，发送消息，极易出现漏发、错发
- **追踪效率低下**：员工回复散落在各个聊天窗口，需人工逐一确认
- **数据一致性风险**：考勤各类审批中的数据缺乏交叉校验，容易出现确认后又增加审批通过数据导致源数据错误
- **进度不透明**：管理者无法实时掌握整体确认进度

---

## 三、解决方案（Action）

### 3.1 技术架构

```
签字报表(Excel)
    ↓
┌─────────────────────────────────────────────┐
│  脚本自动化流程                              │
├─────────────────────────────────────────────┤
│  ① 初始化状态表 → 生成确认状态.csv           │
│  ② 查询open_id → 飞书用户ID映射             │
│  ③ 发送确认消息 → 飞书卡片消息（304人）      │
│  ④ 轮询接收回复 → 常驻进程实时接收           │
│  ⑤ 生成进度报告 → HTML可视化看板            │
│  ⑥ 上传多维表格 → Bitable同步              │
│  ⑦ 数据校验 → 交叉校验5类数据               │
└─────────────────────────────────────────────┘
```

### 3.2 核心功能模块

| 模块        | 功能说明                       | 技术实现           |
| --------- | -------------------------- | -------------- |
| 初始化状态表    | 生成确认状态追踪表                  | Python + CSV   |
| 查询open_id | 批量查询员工飞书open_id            | 飞书通讯录API       |
| 发送确认消息    | 飞书卡片消息（含REG/OT1.5/OT2.0数据） | 飞书消息API        |
| 轮询常驻进程    | launchd托管，实时接收员工回复         | feishu_poll.py |
| 进度报告      | HTML可视化确认进度                | HTML + CSS     |
| 多维表格      | Bitable实时同步确认状态            | 飞书Bitable API  |
| 数据校验工具    | 5类校验规则交叉验证                 | Python校验引擎     |

### 3.3 数据校验规则

| 规则 | 校验内容 | 发现问题数 |
|------|----------|------------|
| R1.2 | 同一日期重复打卡（间隔<10分钟） | 59条 |
| R3 | OT时长偏差超5% | 211条 |
| R4 | 补签审批中但REG=0 | 36条 |
| R5 | REG为零但有打卡记录 | 259条 |
| **合计** | | **565条异常** |
